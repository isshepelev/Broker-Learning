# Проект для ППР
# Broker Learning — интерактивная платформа для изучения Apache Kafka

Веб-приложение для практического изучения Apache Kafka, сравнения с RabbitMQ и демонстрации ключевых концепций брокеров сообщений.

## Технологии

- **Backend:** Spring Boot 4.0.3, Spring Kafka, Spring AMQP
- **Frontend:** Thymeleaf, Bootstrap 5.3.3
- **Брокеры:** Apache Kafka, RabbitMQ
- **БД:** PostgreSQL 16
- **Инфраструктура:** Docker Compose

## Быстрый старт

```bash
# Запуск всей инфраструктуры (Kafka, RabbitMQ, PostgreSQL, Kafka UI)
docker compose up -d

# Запуск приложения
./mvnw spring-boot:run
```

Приложение будет доступно по адресу: **http://localhost:8080**

### Внешние UI

| Сервис | URL |
|--------|-----|
| Kafka UI | http://localhost:8090 |
| RabbitMQ Management | http://localhost:15672 (guest / guest) |

## Разделы приложения

### 1. Dashboard (`/`)

Главная страница с общей информацией о системе.

**Что показывает:**
- Количество топиков в кластере
- Количество последних сообщений
- Статус Producer/Consumer джобов (запущен/остановлен)
- Список топиков в кластере (кликабельные бейджи)
- Таблица последних сообщений (топик, ключ, значение, направление, время)

---

### 2. Producer (`/producer`)

Отправка сообщений в Kafka с выбором стратегии.

**Возможности:**
- Выбор топика из списка существующих
- Ввод ключа сообщения (опционально)
- Ввод значения сообщения (JSON или текст)
- Указание партиции (опционально, иначе авто-назначение)
- Добавление кастомных заголовков (headers)

**Режимы отправки:**
| Режим | Описание |
|-------|----------|
| **Fire-and-Forget** | Отправка без ожидания подтверждения. Самый быстрый, но может потерять сообщения |
| **Sync** | Синхронная отправка с ожиданием подтверждения от брокера |
| **Callback** | Асинхронная отправка с callback-уведомлением о результате |
| **Transactional** | Отправка внутри Kafka-транзакции. Гарантия exactly-once |

**Как пользоваться:**
1. Выберите топик из выпадающего списка
2. Введите ключ и значение сообщения
3. Выберите режим отправки
4. Нажмите «Отправить» — результат отобразится внизу

---

### 3. Consumer (`/consumer`)

Просмотр полученных сообщений из Kafka в реальном времени.

**Что показывает:**
- Таблица сообщений: топик, ключ, значение, партиция, offset, время
- Счётчик сообщений и время последнего обновления
- Авто-обновление каждые 2 секунды

---

### 4. Topics (`/topics`)

Управление Kafka-топиками.

**Возможности:**
- **Создание топика:** указание имени, количества партиций (1-100), фактора репликации (1-10)
- **Просмотр списка** всех существующих топиков
- **Детали топика** (`/topics/{name}`): конфигурация в формате ключ-значение
- **Удаление топика** с подтверждением

---

### 5. Partitions (`/partitions`)

Визуализация распределения сообщений по партициям.

**Как пользоваться:**
1. Выберите топик
2. Введите ключи через запятую (или используйте пресеты: «3 пользователя», «5 заказов», «8 сенсоров», «без ключа — round-robin»)
3. Укажите количество сообщений на ключ (1-50)
4. Нажмите «Отправить»

**Что демонстрирует:**
- Сообщения с одинаковым ключом **всегда** попадают в одну партицию (хеш-распределение)
- Без ключа — round-robin распределение
- Группировка сообщений по партициям с визуализацией

---

### 6. Семантики доставки (`/delivery-semantics`)

Демонстрация трёх уровней гарантий доставки сообщений.

| Семантика | Конфигурация | Гарантия |
|-----------|-------------|----------|
| **At-Most-Once** | `acks=0`, auto-commit, без retry | Может потерять, не дублирует |
| **At-Least-Once** | `acks=all`, manual commit, retry | Не потеряет, может дублировать |
| **Exactly-Once** | Транзакции, идемпотентный producer, `read_committed` | Ровно одна доставка |

**Как пользоваться:**
- Нажмите кнопку «Запустить демо» в каждой карточке
- Укажите количество сообщений для теста
- Результат покажет поведение: сколько отправлено, сколько получено, были ли потери или дубли

---

### 7. Jobs (`/jobs`)

Управление фоновыми задачами Producer и Consumer.

**Producer Job:**
- Отправляет JSON-метрики каждые 3 секунды в `metrics-topic`
- Кнопки Start/Stop
- Лог отправленных сообщений в реальном времени

**Consumer Job:**
- Подписан на `metrics-topic`
- Обрабатывает входящие сообщения
- Кнопки Start/Stop
- Лог полученных сообщений в реальном времени

**Как пользоваться:**
1. Нажмите «Запустить» для Producer — он начнёт генерировать метрики
2. Нажмите «Запустить» для Consumer — он начнёт читать метрики
3. Наблюдайте за логами обоих в реальном времени

---

### 8. Consumer Groups (`/consumer-groups`)

Информация об архитектуре consumer-групп.

**Что показывает:**
- Список всех consumer-групп в кластере
- Образовательные карточки:
  - Правила назначения партиций
  - Стратегии ребалансировки (eager vs cooperative)
  - Управление offset'ами

**Ключевые концепции:**
- Каждая партиция назначается ровно одному consumer'у в группе
- Один consumer может обрабатывать несколько партиций
- При изменении состава группы происходит ребалансировка

---

### 9. История (`/history`)

Полная история всех сообщений с поиском и фильтрацией.

**Возможности:**
- Поиск по топику, ключу или значению
- Фильтр по направлению: Все / Отправленные / Полученные
- Сортировка по любому столбцу
- Данные хранятся в PostgreSQL

---

### 10. Replay (`/replay`)

Демонстрация ключевого преимущества Kafka — возможности перечитывать сообщения.

**Элементы интерфейса:**
- **Визуальная шкала offset'а:** зелёная часть — прочитанные, оранжевая — непрочитанные
- **Поле ввода** для отправки сообщений с произвольным текстом
- **Кнопки управления** (в одну строку):
  - «Прочитать» — читает сообщения с текущей позиции
  - «Сбросить на 0» — сброс offset'а на начало
  - Ввод произвольного offset'а
  - «Очистить топик» — удаление всех сообщений

**Как пользоваться:**
1. Отправьте несколько сообщений (свои или 5 тестовых)
2. Нажмите «Прочитать» — сообщения появятся в логе
3. Нажмите «Сбросить на 0» — offset вернётся в начало
4. Нажмите «Прочитать» снова — те же сообщения прочитаются заново

**Что демонстрирует:** В отличие от RabbitMQ, сообщения в Kafka не удаляются после чтения. Можно перечитать данные за любой период, просто сбросив offset consumer-группы.

---

### 11. Kafka vs RabbitMQ (`/compare`)

Живое сравнение двух брокеров сообщений бок о бок.

**Что показывает:**
- Архитектурные схемы обоих брокеров
- Отправка одного сообщения в оба брокера одновременно
- Разница в поведении:
  - **Kafka:** сообщение остаётся в топике после чтения
  - **RabbitMQ:** сообщение удаляется из очереди после acknowledge

**Сравнительная таблица (11 параметров):**
- Модель данных (лог vs очередь)
- Хранение сообщений
- Скорость
- Маршрутизация
- Consumer-группы
- Масштабирование
- и другие

**Как пользоваться:**
1. Введите сообщение и нажмите «Отправить в оба»
2. Наблюдайте за логами обоих брокеров
3. Обратите внимание: в Kafka сообщения накапливаются, в RabbitMQ — исчезают после обработки

---

### 12. DLQ Compare (`/dlq-compare`)

Сравнение обработки ошибок: Kafka DLT vs RabbitMQ DLX.

**Kafka DLT (Dead Letter Topic):**
- `DefaultErrorHandler` перехватывает исключение
- 3 попытки retry с интервалом 1 секунда
- После исчерпания retry → сообщение в `.DLT` топик
- Сообщение сохраняется навсегда, можно перечитать

**RabbitMQ DLX (Dead Letter Exchange):**
- Consumer бросает `AmqpRejectAndDontRequeueException`
- Мгновенный reject без retry
- Сообщение попадает в Dead Letter Exchange → Dead Letter Queue
- Быстрее, но нет повторных попыток

**Триггерные слова для ошибки:** `poison`, `bad`, `error`, `fail`, `crash`, `broken`

**Элементы интерфейса:**
- Схемы потоков для каждого брокера
- Ввод сообщения + кнопка «Отправить poison»
- Логи событий бок о бок (SENT → RETRY → DLT / SENT → FAIL → DLQ)
- Таблицы «мёртвых» сообщений (Kafka DLT и RabbitMQ DLQ)
- Сравнительная таблица ключевых отличий

**Как пользоваться:**
1. Нажмите «Отправить poison» — «плохое» сообщение отправится в оба брокера
2. В Kafka: наблюдайте 4 RETRY (отправка + 3 повтора) → DLT (~4 сек)
3. В RabbitMQ: FAIL → DLQ мгновенно
4. В секции «мёртвые сообщения» отобразятся оба сообщения

---

### 13. Benchmark (`/benchmark`)

Сравнение производительности четырёх режимов отправки.

**Режимы:**
| Режим | Описание |
|-------|----------|
| **Fire-and-Forget** | Отправка без подтверждения — максимальная скорость |
| **Sync** | Синхронное ожидание каждого подтверждения — минимальная скорость |
| **Async/Callback** | Все отправки сразу, потом ожидание подтверждений — баланс |
| **Transactional** | Все сообщения в одной транзакции — гарантия exactly-once |

**Как пользоваться:**
1. Выберите количество сообщений (10-10 000) или используйте пресеты (100, 500, 1000, 5000)
2. Нажмите «Запустить» — все 4 режима будут протестированы последовательно
3. Результаты отобразятся в виде:
   - Графика времени выполнения (мс)
   - Графика пропускной способности (сообщений/сек)
   - Карточек с описанием каждого режима
   - Детальной таблицы результатов

---

### 14. Мониторинг (`/monitoring`)

Мониторинг кластера в реальном времени.

**Что показывает:**
- **Карточки статистики:** всего отправлено, всего получено, за последнюю минуту, за 5 минут
- **График пропускной способности:** сообщений за 10 секунд, скользящее окно 5 минут
- **Consumer Group Lag:** отставание каждой consumer-группы (помогает найти медленных потребителей)
- **Размеры топиков:** количество сообщений в каждом топике

Все данные обновляются автоматически каждые 3 секунды.

## Конфигурация

### application.yaml

| Параметр | Значение | Описание |
|----------|----------|----------|
| `spring.kafka.bootstrap-servers` | `localhost:9092` | Адрес Kafka брокера |
| `spring.kafka.producer.acks` | `all` | Подтверждение от всех реплик |
| `spring.kafka.consumer.auto-offset-reset` | `earliest` | Чтение с начала при отсутствии offset'а |
| `spring.kafka.consumer.enable-auto-commit` | `false` | Ручное управление commit'ами |
| `spring.kafka.listener.ack-mode` | `manual` | Ручной acknowledge |
| `spring.rabbitmq.host` | `localhost` | Адрес RabbitMQ |
| `spring.datasource.url` | `jdbc:postgresql://localhost:5432/broker_learning` | PostgreSQL |
